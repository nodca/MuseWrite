# Single Workspace Unification 设计稿

日期：2026-03-11

## 背景

当前 Web 端已经形成两套产品叙事：

- 写作模式：正文优先，尽量少打扰。
- 工作台模式：暴露动作提议、Prompt、规划、evidence、检索快照。

但从代码和交互看，这两者并不是两套独立产品，而是同一套能力的不同密度外壳：

- 核心状态共用同一个 store，`uiMode` 主要控制壳层切换。
- 主要复杂度集中在 `apps/web/src/App.tsx` 的条件渲染与面板编排。
- 作者手册已经默认把写作模式视为主路径，工作台只是补充说明。

这说明当前问题不是“功能太多”，而是“功能暴露方式裂成了两种心智模型”。用户必须先理解“我现在在哪个模式”，才能理解“我能做什么”，这会持续抬高认知成本。

## 目标

- 收束为始终只有一个创作工作区，不再让用户切换“写作模式 / 工作台模式”。
- 保留现有高价值能力，但改为按需展开，而不是整页切换。
- 让正文、章节、大纲、动作提议、Prompt、evidence/debug 形成一条连续工作流。
- 继续照顾新手默认路径，同时保留专家用户对高级能力的进入方式。
- 为后续拆解 `App.tsx`、删除 `uiMode`、清理死代码创造清晰边界。

## 非目标

- 本次不改写聊天/检索/动作闭环后端协议。
- 本次不重新定义章节、Beat、伏笔、Prompt 的领域模型。
- 本次不立即删除后端所有底层 env，只先完成“用户心智简化”和“前台暴露收敛”。
- 本次不做全新视觉语言，只在现有视觉基础上重组信息架构。

## 现状问题

### 1. 模式切换制造了额外心智负担

用户在“写作”和“分析”之间来回跳，实际上是在切 UI 壳，而不是切任务上下文。

### 2. 高级能力被整体打包，缺乏渐进暴露

当前工作台能力是整包出现的，导致：

- 新手会回避它。
- 老手必须频繁切换。
- 具体能力之间无法独立显隐。

### 3. `App.tsx` 已经成为编排中心

页面主壳、顶部操作、设置弹窗、助手抽屉、写作区、工作台区都压在一个入口里。继续在“双模式”上修补只会放大复杂度。

### 4. 配置分层已经有理念，但还没有落成统一心智

前台已有“基础写作 / 模型与检索 / 高级调优”的收敛趋势，README 也强调 `profile + 少量策略开关`，但产品层和工程层还没有被明确切开。

## 方案对比

### 方案 A：保留双模式，只弱化文案

把“工作台模式”改成“高级模式”或“分析视图”，保留整体切换。

优点：

- 实现最快。
- 改动范围相对小。

缺点：

- 仍然保留双心智。
- 只是改名，不是收敛。

### 方案 B：单一工作区 + 渐进展开

始终保留一个主工作区，正文是中心，其他能力按面板和侧栏按需展开。

优点：

- 真正消灭模式心智。
- 保留专家能力，不牺牲新手路径。
- 便于后续拆分壳层组件。

缺点：

- 需要较大前端壳层重构。
- 需要重新定义哪些能力默认显示。

### 方案 C：按任务阶段重构整站

把整个产品改成“写作 / 规划 / 审阅 / 调优”四个阶段入口。

优点：

- 概念上最完整。

缺点：

- 改动过大。
- 会打断现有高频闭环。

## 定案

采用方案 B：单一工作区 + 渐进展开。

核心原则：

1. 正文永远是主舞台，不能再被“切模式”掩盖。
2. 进阶能力应是“展开更多信息”，而不是“跳到另一页人格”。
3. 能力按任务邻近性组织，而不是按实现来源组织。
4. 默认界面必须足够轻；高级能力必须足够近。

## 信息架构

统一工作区拆成四个稳定区域。

### 1. 顶部控制条

保留：

- 写作设置
- 助手入口
- 新会话
- 刷新快照
- 专注写作 / 沉浸切换
- 进阶面板展开按钮

删除：

- 模式切换按钮

顶部控制条只负责“动作入口”，不再承担“页面人格切换”。

### 2. 中央主舞台

中央区域固定为正文工作区，继续承载：

- 章节切换
- 正文编辑
- 自动保存状态
- 选区润色/扩写
- 助手回复写回正文
- 版本历史

这部分沿用当前 `DraftWorkspacePanel` 的核心能力，但需要从“写作模式组件”中解耦出来，成为统一壳的中心主体。

### 3. 左侧导航区

左侧保留轻量级结构导航，优先承载：

- 章节大纲/切章
- 当前卷绑定信息
- 基础写作状态摘要

目标是让结构导航和正文天然相邻，不必切到“工作台”才能看见。

### 4. 右侧进阶辅助栏

右侧是唯一的“渐进展开容器”，默认可折叠。内部采用可独立展开/收起的 section：

- 结构化规划
- 动作提议与审计
- Prompt / 知识注入
- Context / evidence / debug

默认策略：

- 初次进入只展开最常用 section。
- 用户可 Pin 常用 section。
- debug/evidence 只在用户主动展开时出现。

## 交互策略

### 默认态

- 页面直接进入统一工作区。
- 中央正文 + 左侧结构导航可见。
- 右侧进阶辅助栏默认关闭或仅展开一个常用 section。

### 进阶展开态

- 用户点击“展开进阶面板”后，右侧辅助栏打开。
- 不改变正文位置语义，只改变可见信息密度。
- 用户关闭某个 section 时，不影响其他 section。

### 专注写作态

- 保留 `zenMode / focus mode`，但它只是一种写作沉浸状态，不是新模式。
- 专注态可以自动折叠右侧辅助栏和非必要顶部控件。

## 状态模型

需要删除 `uiMode` 这一状态概念，改成更贴近真实意图的壳层状态：

- `workspaceInspectorOpen: boolean`
- `workspaceSections: { planning, actions, prompt, context }`
- `workspacePinnedSection: ... | null`
- `zenMode: boolean`

说明：

- `zenMode` 继续表示沉浸写作。
- `workspaceInspectorOpen` 表示右侧辅助栏是否展开。
- 各 section 独立控制显隐，代替“工作台模式”整体可见。

这会让状态含义从“我在哪个模式”变成“我现在展开了哪些工具”。

## 组件重组

建议的新壳层结构：

- `CreativeWorkspaceShell`
- `WorkspaceTopBar`
- `WorkspaceInspectorRail`
- `WorkspaceSectionTabs` 或 `WorkspaceSectionStack`
- `DraftEditorPane`

现有组件处置建议：

- `ModeSwitch.tsx`：删除。
- `WritingWorkspaceMode.tsx`：删除，正文能力下沉到统一壳。
- `ProWorkspaceMode.tsx`：删除，工作台面板拆为 rail sections。
- `DraftWorkspacePanel.tsx`：保留核心编辑能力，但改造成统一壳里的中央 pane。

## 配置收敛策略

### 产品层

作者可感知的配置只保留两层：

- 基础写作
- 进阶设置

其中“进阶设置”仍可分组，但不再以“懂技术才能改”的方式暴露。

### 工程层

后端继续保留 profile 与 runtime defaults，但前台和文档应清晰区分三类配置：

- 作者可调：写作主题、模型 profile、写作偏好、少量策略
- 项目可调：跨项目引用、章节上下文偏好
- 部署可调：provider、timeout、cache、circuit breaker、queue、GDS/LightRAG

首轮重构不要求删除全部 env；要求先停止把部署细节投射成产品心智。

## 迁移路径

### Phase 1：壳层统一

- 引入统一工作区壳组件
- 删除顶部模式切换
- 把工作台面板收编进右侧辅助栏

### Phase 2：状态与死代码清理

- 删除 `uiMode`
- 删除旧模式组件
- 更新对应测试与文档

### Phase 3：配置与文案收敛

- 设置弹窗重写为“基础 + 进阶”
- README / 作者手册同步为单一工作区叙事

## 风险

### 1. 高级用户担心能力被藏深

应对：

- 允许 Pin 常用 section。
- 为右侧辅助栏提供明确入口与记忆状态。

### 2. `App.tsx` 重构期间容易引入回归

应对：

- 先补壳层 E2E，再做组件迁移。
- 每一步只移动一类职责。

### 3. “沉浸写作”与“辅助栏展开”冲突

应对：

- 明确两者不是互斥模式。
- 专注写作优先级高于辅助栏展开。

## 验收标准

- 顶部不再出现“写作模式 / 工作台模式”切换。
- 用户首次进入即可完成正文写作，不必理解“模式”。
- 高级能力能在同一页面按需展开。
- `uiMode` 从 store、组件、E2E、文档中被彻底删除。
- README 与作者手册不再使用双模式叙事。
