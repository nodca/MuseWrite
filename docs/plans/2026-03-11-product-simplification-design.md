# Novel Platform 产品精简与概念收敛设计稿

日期：2026-03-11

## 背景

当前项目的问题已经不只是“双模式”。

真正的复杂度来自四层叠加：

- 同一用户意图被拆成多个产品概念，例如 `GhostText`、灵感建议、写作助手、工作台、改写、规划。
- 前端壳层与状态编排集中在 `apps/web/src/App.tsx`，模式、助手、Ghost、规划、调试彼此缠绕。
- 后端把大量实现自由度直接映射成前台概念，例如 `rag_mode`、`thinking_enabled`、`context_window_profile`、`model_profile_id`。
- 文档、测试、历史计划仍把“写作模式 / 工作台模式 / Ghost Text / 灵感工作台”视为既定主路径。

这意味着当前要解决的核心，不是“再做一个更好的模式切换”，而是“让产品重新只围绕极少数真实意图组织”。

## 目标

- 把首屏收束为纯编辑器优先的新手写作体验。
- 取消“写作模式 / 工作台模式”双心智。
- 明确产品只围绕 `写作`、`规划`、`高级` 三层组织。
- 删除 `GhostText` 与 `重写` 这类已不符合主方向的能力概念。
- 让“写作助手”退回到单一职责：默认帮助规划，而不是成为功能杂物柜。
- 为后续前端拆壳、配置收口、测试重写建立可执行边界。

## 非目标

- 本次不立即重写后端检索、上下文编排、图谱、一致性巡检等底层实现。
- 本次不立即删除所有 Ghost 相关后端兼容接口。
- 本次不重新定义章节、卷、Beat、伏笔等领域模型。
- 本次不先做视觉重设计，重点是信息架构与概念收敛。

## 核心判断

### 1. 产品真正的一级意图只有三个

- `写作`：直接编辑正文、管理章节、保存版本。
- `规划`：思考章节推进、Beat、伏笔、设定与后续剧情。
- `高级`：查看 Prompt、evidence、动作审计、图谱检视等专家能力。

除此之外的大部分名词，不是一级能力，只是触发方式、实现方式或历史包装。

### 2. “写作助手”不应该是全能助手

根据当前方向，`写作助手抽屉` 的默认职责是“帮助规划”，而不是“承载所有 AI 能力”。

因此：

- 规划属于抽屉主路径。
- 改写不属于抽屉主路径。
- Prompt、调试、evidence 不属于默认抽屉首页。

### 3. 改写应回归正文就地操作

现有“润色 / 扩写”已经有更自然的触发方式：选中正文后右键。

这比把改写放进抽屉更符合写作流，因此：

- 保留：润色、扩写。
- 删除：重写。
- 删除：GhostText。
- 抽屉不再承担改写入口。

### 4. “工作台”是历史容器，不是未来概念

工作台当前只是把规划、Prompt、调试、快照、动作提议重新打成一个“第二人格”。

它没有稳定的用户意图，因此应删除这个产品概念，而不是继续改名续命。

## 新产品骨架

### 首屏默认态

首屏默认只保留 5 个元素：

- 左侧极窄章节列表
- 中央正文编辑器
- 顶部极简控制条
- 右下角助手入口
- 右侧中部单个悬浮按钮

用户第一次进入时，不需要理解任何“模式”“工作台”“Ghost”“上下文策略”，只需要开始写。

### 写作助手入口

右侧中部悬浮按钮是单个入口，不是多按钮列。

建议命名：

- `写作助手`
- 或更聚焦的 `剧情助手`

不再使用：

- 工作台
- 专业模式
- 分析模式

### 写作助手抽屉

点击悬浮按钮后，打开 `写作助手抽屉`。

抽屉默认落在 `规划` 首页，一级只保留三组内容：

1. 章节推进
2. 情节规划
3. 设定伏笔

这三组可由当前已有的 Volume / Scene Beat / Foreshadow / 设定卡能力重组而来。

### 改写入口

改写只存在于正文就地交互中：

- 选区右键 `润色`
- 选区右键 `扩写`
- 助手回复写回正文 `插入 / 替换`

改写不再出现在写作助手抽屉首页，也不再作为一级产品栏目。

### 高级区

高级区单独折叠，不进入新手默认路径。

包含：

- Prompt 模板与知识注入
- 动作提议与审计
- evidence / Context X-Ray / 检索快照
- 图谱候选审查、时间线、Blast Radius 等专家能力

高级区仍然保留，但它们的职责从“第二主界面”变为“必要时打开的深层检视”。

## 功能收敛决策

### 保留

- 正文编辑
- 自动保存 / 崩溃恢复 / 版本回滚
- 章节创建、切换、排序
- 选区润色
- 选区扩写
- 助手回复插入 / 替换正文
- 章节规划、卷纲、Beat、伏笔
- 设定卡与提示模板
- evidence / 动作审计 / 图谱高级能力

### 合并或降级

- `灵感建议`
  - 若本质是后续剧情建议，合并进规划助手。
  - 若本质是续写提示，不再作为一级概念。
- `写作助手`
  - 保留为容器名，不再代表全能 AI 平台。
- `awareness strip`
  - 降级为轻量认知提示，避免继续膨胀为首屏调试面板。
- `工作台`
  - 仅保留为迁移期内部名词，产品层删除。

### 删除

- `写作模式 / 工作台模式`
- `uiMode`
- `ModeSwitch`
- `GhostText`
- `重写`

## 架构影响

### 前端

前端需要做三类拆分：

1. 壳层拆分
   - 让 `App.tsx` 从“全功能总集线器”退回页面编排器。
2. 概念拆分
   - 把“规划”“改写”“高级”分别收束到单一入口。
3. 状态拆分
   - 删除 `uiMode`，改成“抽屉是否打开”“高级区是否展开”“当前助手页签”等显式状态。

### 后端

后端短期不需要先删服务，但需要明确边界：

- 保留 story workspace CRUD 主干接口。
- 保留 chat stream 主干接口。
- 保留高级能力所需的 actions / evidence / graph / audit 接口。
- 前端先停止暴露 `rag_mode`、`thinking_enabled`、`enable_tot`、`context_window_profile` 等高级控制。
- 这些字段优先收回到 profile/defaults/server heuristics。

### 文档与测试

必须同步更新：

- 作者手册
- README 主叙事
- E2E 冒烟用例
- Ghost 相关测试
- 历史计划文档状态说明

否则旧概念会持续通过测试与文档反向塑形产品。

## 分阶段迁移

### Phase 1：概念清算

- 明确保留、合并、删除列表。
- 冻结新增工作台/Ghost/灵感类概念。
- 先更新设计与实施计划。

### Phase 2：壳层统一

- 删除模式切换。
- 建立纯编辑器首页。
- 接入单个悬浮按钮与写作助手抽屉。

### Phase 3：能力重定位

- 规划能力迁入抽屉首页。
- 改写能力收回选区工具。
- 高级能力收到底层折叠区。

### Phase 4：Ghost 与旧契约清除

- 删除 Ghost UI、状态、测试。
- 清理 `uiMode` 及其相关条件分支。
- 重写 E2E 与作者手册叙事。

### Phase 5：后端与配置收口

- 收缩前台可见配置项。
- 保持协议兼容，逐步去除不再暴露的前台参数。
- 视迁移结果决定 Ghost 兼容层是否继续保留。

## 风险

### 1. 只改名，不减复杂度

如果只是把“工作台”改叫“助手抽屉”，但 Ghost、规划、调试、Prompt、awareness 仍各自为政，复杂度几乎不变。

### 2. 只删前台，不清测试契约

若 `smoke.spec.ts`、`blast-radius-preview.spec.ts`、Ghost 相关测试不先迁移，重构会一直被旧断言拖住。

### 3. 误删底层协议

应先停止暴露前台高级概念，再考虑是否缩减后端字段与接口，避免在重构 UI 时打断已有主流程。

## 验收标准

- 首屏不再出现模式切换。
- 新用户进入后无需理解工作台/Ghost 即可开始写作。
- 写作助手抽屉默认服务规划。
- 改写只存在于正文就地操作。
- `GhostText`、`重写`、`工作台模式` 不再属于产品主叙事。
- `uiMode` 从前端代码、测试、文档中删除。
- 高级能力仍可访问，但默认下沉。
